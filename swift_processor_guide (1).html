Section 01

Vue d'ensemble

Le SWIFT Processor automatise le rapprochement entre les ordres de paiement (MX103/PACS.008) et les avis de crédit (MT910) circulant sur le réseau SWIFT. Les messages sont extraits depuis des fichiers PDF déposés sur deux lecteurs réseau distincts.

Objectifs métier

  • Identifier les paires (PACS.008 ↔ MT910) qui correspondent à la même opération bancaire.
  • Classer physiquement les fichiers matchés dans une arborescence MATCH/MMAA/JJMMAA/ avec un identifiant commun.
  • Isoler les fichiers non matchés depuis plus de 10 jours dans PAS_MATCH/ pour investigation manuelle.
  • Traiter en continu les nouveaux fichiers déposés au fil de l'eau (cron toutes les 10 minutes).

Principes de conception

Conception

Incrémental — chaque fichier est traité une seule fois, même sur des dizaines de relances.

Idempotent — un crash en cours d'exécution n'altère pas les données ; la relance reprend proprement.

Économe en I/O — élagage agressif des sous-dossiers anciens ou hors-périmètre pour limiter les appels réseau.

Auditable — chaque décision (match, skip, erreur de lecture) est tracée en CSV et dans les logs.

Section 02

Architecture des modules

Le projet est découpé en six modules à responsabilité unique. Chaque fichier traite une préoccupation distincte du pipeline.

01
config.py
Centralise les chemins, constantes métier (START_DATE, seuils), filtres de sous-dossiers et paramètres de logging.
02
storage.py
Gère les 5 CSV de persistance (MX103, MT910, matches, copies, NO_read). Lecture forcée en str pour éviter la corruption silencieuse.
03
pdf_extractor.py
Extrait les champs métier (montant, date, debtor, creditor, référence) depuis les PDF avec PyMuPDF et regex robustes.
04
matcher.py
Implémente le matching séquentiel en deux étapes (référence puis cle_32A+debtor+creditor) avec match_id stables.
05
file_manager.py
Indexe les fichiers source avec élagage par ctime et filtres de chemins. Copie les fichiers vers MATCH/ et PAS_MATCH/.
06
main.py
Point d'entrée. Orchestre le pipeline complet : init, indexation, extraction, matching, copies.

Section 04

Configuration centrale

Toute la configuration vit dans config.py. C'est le seul fichier à ajuster pour changer d'environnement.

Chemins

# Chemins de production
MX103_PATH = 'Y:/'
MT910_PATH = 'Z:/SWIFT_SGCI'
OUTPUT_PATH = 'Y:/TRAITEMENT_MX_MT910'

Filtres de sous-dossiers

Chaque filtre est une liste de segments de chemin. Le caractère * agit comme wildcard sur un segment unique.

MX103_SUBDIR_FILTERS = [
    ["entrant", "pacs.008", "*", "auto"],
    ["entrant", "pacs.008", "*", "manu", "sgci"],
]

MT910_SUBDIR_FILTERS = [
    ["entrant", "mt910"],
]
Matching par segments

Les termes sont comparés segment par segment (pas par sous-chaîne). Ainsi auto ne matche pas 2026_01_auto ni un nom de fichier file_auto.pdf.

Paramètres métier

ParamètreValeur par défautDescription
START_DATE datetime(2026, 1, 1) Date à partir de laquelle les dossiers MMAA sont indexés. Les dossiers plus anciens sont élagués.
DAYS_THRESHOLD_NO_MATCH 10 Nombre de jours après lesquels un fichier non matché part en PAS_MATCH/.
MX103_MAX_PAGES 2 Pages PDF à lire au maximum pour les MX103.
MT910_MAX_PAGES 1 Pages PDF à lire au maximum pour les MT910.
Section 05

Flux de traitement

Chaque exécution suit cinq étapes séquentielles. L'incrémentalité garantit qu'un fichier déjà traité ne sera pas retraité.

01
Initialisation
Création des dossiers de sortie, initialisation des 5 CSV s'ils n'existent pas, chargement des sets en mémoire (NO_read, copied).
02
Indexation des sources
Parcours de Y:/ puis Z:/ avec élagage des sous-dossiers anciens (ctime < START_DATE) et hors filtre. Construction d'un dict {filename: filepath}.
03
Extraction PDF
Pour chaque nouveau fichier de l'index (non présent dans les CSV), extraction des champs métier avec PyMuPDF. Les PDF illisibles sont enregistrés dans NO_read.
04
Matching
Logique séquentielle : Étape 1 par référence (filename split par _), Étape 2 par cle_32A + debtor + creditor inclus dans texte_72. Les match_id sont stables entre runs.
05
Copie physique
Les paires matchées sont copiées dans MATCH/MMAA/JJMMAA/ avec un préfixe match_id_. Les fichiers non matchés depuis >10 jours partent dans PAS_MATCH/.

Robustesse

Les PDF qui échouent à l'extraction sont enregistrés dans no_read_files.csv et ne seront plus retentés aux runs suivants. Cette décision permet d'éviter de bloquer le pipeline sur un PDF corrompu.

Section 07

Logique de matching

Le matching s'effectue en deux étapes séquentielles. Les paires trouvées à l'étape 1 sont retirées des pools avant l'étape 2 pour éviter tout double comptage.

Étape 1 — Par référence

Jointure exacte sur les références extraites des noms de fichiers :

# reference_mx (4e segment du nom PACS) == reference_mt (5e segment du nom MT)
reference_mx == reference_mt  # comparaison lowercase + trim

Les paires trouvées reçoivent match_type = "reference". Les fichiers MX et MT impliqués sont retirés des pools avant l'étape 2.

Étape 2 — Par montant + acteurs

Sur le pool restant, jointure sur cle_32A puis filtre :

cle_32A_mx == cle_32A_mt
AND adresse_debtor in texte_72
AND adresse_creditor in texte_72

Les paires trouvées reçoivent match_type = "32A_debtor".

Match IDs stables

Les match_id sont des entiers incrémentaux conservés entre les runs. Une paire (cle_32A, filename_mx, filename_mt) conserve son ID à vie. Cela garantit l'idempotence en cas de relance après interruption.

Protections contre les vides

  • Les cle_32A vides sont exclus du merge (évite le produit cartésien massif).
  • Les reference vides sont exclues du merge de l'étape 1.
  • Si debtor, creditor ou texte_72 est vide → pas de match (protection contre "" in texte).
Section 08

Stockage CSV

Cinq fichiers CSV constituent l'état persistant du système. Ils permettent l'exécution incrémentale et l'audit.

FichierRôleClé naturelle
mx103_files.csv Cache des MX103 déjà extraits filename
mt910_files.csv Cache des MT910 déjà extraits filename
matches.csv Paires (MX, MT) matchées (cle_32A, filename_mx, filename_mt)
copied_files_log.csv Log des fichiers déjà copiés (anti-doublon) filepath de destination
no_read_files.csv Log des PDF illisibles (à skipper) filepath source
Lecture en chaîne

Tous les CSV sont lus avec dtype=str et .fillna("") pour éviter que pandas ne convertisse silencieusement les chiffres en float (ce qui produisait l'ancien bug 123456.0).

Section 09

Filtrage & performance

Sur lecteurs réseau, chaque appel os.path.* coûte cher. Le système applique trois niveaux d'élagage pour minimiser les appels.

Niveau 1 — Élagage par ctime (MMAA uniquement)

Au premier niveau sous Y:/ (les dossiers MMAA comme 0126, 0226...), le système vérifie ctime >= START_DATE. Les dossiers plus anciens sont entièrement skippés.

Pourquoi seulement MMAA

Les sous-dossiers plus profonds (entrant/, pacs.008/) peuvent avoir un ctime ancien même s'ils contiennent des fichiers récents. Le filtre est appliqué uniquement au niveau MMAA car ces dossiers sont rigidement liés à une période temporelle.

Niveau 2 — Élagage par filtre de chemin

Les sous-dossiers qui ne peuvent pas mener à un chemin matchant un filtre sont skippés en cours de os.walk via dirs[:].

Niveau 3 — Filtres au niveau fichier

Dans extract_new_pdfs, chaque fichier passe par une cascade :

  1. Extension .pdf ?
  2. Déjà dans mx103_files.csv ou mt910_files.csv ?
  3. Déjà dans no_read_files.csv ?
  4. Chemin matchant un filtre (par segments) ?

Section 12

Maintenance & dépannage

Modifier les filtres de chemins

Éditer config.py. La modification prend effet au prochain run. Les fichiers déjà extraits restent en CSV (pas de purge automatique).

Repartir de zéro

Procédure destructive

Pour reset complet : supprimer le dossier csv_data/ et les dossiers MATCH/ et PAS_MATCH/. Le prochain run re-indexera et re-traitera tout.

Changer le schéma d'un CSV

Si une colonne est ajoutée (ex : adresse_creditor récemment), il faut supprimer le CSV concerné avant de relancer. Sinon les anciennes lignes auront des colonnes manquantes.

Décaler START_DATE

# config.py
START_DATE = datetime(2026, 3, 1)  # nouvelle date

Les dossiers MMAA antérieurs ne seront plus indexés. Les CSV existants ne sont pas purgés (les anciennes entrées restent).

Auditer une paire

  1. Ouvrir matches.csv et chercher le match_id.
  2. Récupérer filename_mx et filename_mt.
  3. Vérifier dans mx103_files.csv et mt910_files.csv les champs sources.
  4. Vérifier la valeur de match_type pour savoir quel critère a déclenché le match.
Section 13

FAQ technique

Que se passe-t-il si je relance le script alors qu'une instance tourne déjà ?

Les deux instances tenteront d'écrire dans les mêmes CSV — risque de corruption. Pour un cron, prévoir un mécanisme de lock (non implémenté actuellement). En attendant, s'assurer que la durée d'un run est largement inférieure à l'intervalle du cron.

Pourquoi les match_id sont-ils stables entre les runs ?

Au démarrage du matching, le code lit matches.csv et construit un dictionnaire {(cle_32A, filename_mx, filename_mt) → match_id}. Une paire déjà connue garde son ID. Les nouvelles paires reçoivent max(IDs existants) + 1.

Que se passe-t-il si un PDF est modifié après extraction ?

Rien : le système ne re-extrait pas un fichier déjà présent dans mx103_files.csv / mt910_files.csv. Si une re-extraction est nécessaire, supprimer manuellement la ligne du CSV (la clé est filename).

Comment auditer combien de paires sont matchées par chaque règle ?

import pandas as pd
df = pd.read_csv('matches.csv')
print(df['match_type'].value_counts())

Le système gère-t-il les noms de fichiers en doublon ?

Dans build_source_index, si deux fichiers ont le même nom dans des dossiers différents, le premier rencontré gagne et un warning est loggé. Pas idéal mais rare en pratique.

Comment ajouter un nouveau type de message (ex : MT940) ?

Il faut : (1) ajouter une fonction d'extraction dans pdf_extractor.py, (2) ajouter un schéma CSV dans storage.py, (3) ajouter les filtres et chemins dans config.py, (4) appeler process_message_type dans main.py et adapter le matcher si nécessaire.
